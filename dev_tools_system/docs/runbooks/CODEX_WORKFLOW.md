# CODEX_WORKFLOW

## Назначение
Codex CLI используется как основной локальный coding agent в Ubuntu/WSL-среде разработки MultiKI.

## Где запускается
Codex запускается из корня репозитория MultiKI:

cd ~/multiki
codex

## Главный принцип
Codex не должен получать свободный запрос без контекста.
Перед работой он должен опираться на:
- AGENTS.md
- текущий issue
- секцию PRD
- task packet
- test contract

## Стандартный цикл работы

1. Выбрать issue.
2. Создать отдельную ветку от master.
3. Подготовить task packet.
4. Запустить Codex в корне репозитория.
5. Дать Codex задачу в формате task packet.
6. Требовать test-first:
   - сначала тест
   - потом код
7. После изменений Codex обязан запустить:
   ./dev_tools_system/scripts/verify.sh
8. Codex показывает:
   - что изменил
   - какие файлы затронул
   - результат verify
   - краткий diff summary
9. Пользователь вручную проверяет diff.
10. Только после этого:
   - git add
   - git commit
   - git push
   - PR
   - CI
   - merge

## Что Codex обязан делать
- читать AGENTS.md
- работать только в рамках текущей задачи
- соблюдать test-first
- запускать verify перед завершением задачи
- показывать итог изменений

## Что Codex не должен делать
- не делать git commit без явной команды
- не делать git push без явной команды
- не менять несвязанные файлы
- не расширять scope задачи
- не принимать архитектурные решения без явного запроса

## Минимальный шаблон запроса к Codex

Контекст:
[кратко]

Источник требований:
PRD section: [название секции]

Issue:
[#номер issue]

Текущая цель:
[что сделать]

Test contract:
- какой тест создать/обновить
- что он должен подтвердить

Ограничения:
- сначала тест
- потом код
- не делать commit
- не менять несвязанные файлы

Definition of Done:
[критерии готовности]

Финал:
- запусти ./dev_tools_system/scripts/verify.sh
- покажи diff summary
- покажи результат verify

## Результат
После этого runbook Codex становится не "чатом для кода", а стандартным исполнителем в системе разработки MultiKI.
