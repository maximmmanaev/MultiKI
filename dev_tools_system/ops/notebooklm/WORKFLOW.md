# NotebookLM Workflow

## Назначение
NotebookLM используется как исследовательский и аналитический слой для системы разработки MultiKI.

## Когда использовать
- когда нужно найти пробелы в PRD
- когда нужно выявить противоречия в документации
- когда нужно разобрать большую пачку документов
- когда нужно получить список рисков
- когда нужно понять, каких тестовых сценариев не хватает

## Когда не использовать
- не использовать как источник истины
- не использовать как систему задач
- не использовать вместо GitHub Project
- не использовать вместо git-репозитория

## Базовые notebook'и
- MultiKI Core Docs
- Sprint Research
- External Tech Docs

## Что загружать
- dev_tools_system/docs/PRD.md
- dev_tools_system/docs/ARCHITECTURE.md
- dev_tools_system/docs/DEV_SYSTEM.md
- dev_tools_system/docs/adr/*
- dev_tools_system/docs/sprints/*
- внешние спецификации и документации

## Базовые вопросы
- где пробелы в PRD?
- какие противоречия есть в документации?
- какие риски есть у текущей архитектуры?
- какие тестовые сценарии следуют из требований?
- какие части MVP определены хуже всего?

## Главное правило
Любой полезный вывод из NotebookLM должен быть перенесён обратно в git:
- в docs/
- в adr/
- в sprint docs
- в issue comments

Нельзя хранить важные решения только внутри NotebookLM.
