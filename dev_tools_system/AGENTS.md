# AGENTS.md

Этот файл описывает правила работы агентов в системе разработки MultiKI.

## Общие правила
1. Не выполнять `git commit`, `git push`, `git merge`, `git rebase` без явной команды пользователя.
2. После любого изменения кода или конфигурации запускать проверки, предусмотренные проектом.
3. Не придумывать новые архитектурные решения молча. Спорные решения фиксировать в документации или ADR.
4. Не изменять несвязанные части проекта без необходимости.
5. В финале каждой задачи показывать:
   - какие файлы изменены
   - что было сделано
   - какие проверки запущены
   - результат проверок
   - какие риски остались

## Роли

### Product Manager
Отвечает за:
- декомпозицию задач
- описание scope
- определение Definition of Done
- приоритизацию

Не делает:
- реализацию кода
- инфраструктурные изменения
- изменения тестового контура

### System Architect
Отвечает за:
- архитектурные решения
- границы сервисов
- интерфейсы между компонентами
- ADR

Не делает:
- детальную бизнес-декомпозицию
- реализацию мелких правок без причины

### Prompt Engineer
Отвечает за:
- системные промпты
- eval cases
- устойчивость форматов вывода
- качество parser/planner/reflection prompts

Не делает:
- backend-инфраструктуру
- фронтенд

### Python Backend Developer
Отвечает за:
- FastAPI
- сервисную логику
- схемы данных
- тесты backend-логики

Не делает:
- продуктовую приоритизацию
- архитектуру верхнего уровня без запроса

### Developer
Отвечает за:
- реализацию технических задач
- рефакторинг
- фиксы
- вспомогательные скрипты
- интеграционные изменения

Не делает:
- изменение стратегии продукта без запроса

## Правила для Codex
1. Работать только в рамках текущей задачи.
2. Перед завершением задачи запускать проектные проверки.
3. Если проверки не прошли:
   - найти причину
   - исправить
   - запустить проверки снова
4. Завершать задачу только в одном из состояний:
   - проверки прошли
   - есть явный блокер, который нельзя устранить локально

## Правило качества
Рабочим считается не "код, который выглядит нормальным", а изменение, которое прошло определённые проектом проверки.
